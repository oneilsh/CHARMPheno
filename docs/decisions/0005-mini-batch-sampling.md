# 0005 — Mini-batch sampling: per-iteration sample-with-replacement, runner-side pre-scaling

**Status:** Accepted
**Date:** 2026-04-29

## Context
Stochastic VI (Hoffman et al., 2013) requires fresh per-iteration sampling for the
Robbins-Monro step-size schedule `(τ₀ + t + 1)^{-κ}` to be principled. Without
sampling, the framework runs full-batch updates on every iteration, the schedule
decays toward zero before convergence, and the framework cannot scale to
production-sized corpora. The bootstrap implementation processed the full RDD
each iteration; this ADR documents the move to mini-batching, modeled on Apache
Spark MLlib's `OnlineLDAOptimizer`.

## Decision
Add three fields to `VIConfig`: `mini_batch_fraction: float | None` (default
`None` = full-batch), `sample_with_replacement: bool` (default `True`), and
`random_seed: int | None` (default `None`). When `mini_batch_fraction` is set,
each iteration of `VIRunner.fit`:

1. Calls `data_rdd.sample(withReplacement, fraction, seed)` to obtain the batch.
2. Persists the batch with `StorageLevel.MEMORY_AND_DISK` so that the
   subsequent `count()` and `mapPartitions` do not recompute the sample.
3. Computes `batch_size = batch.count()`. If zero, the iteration skips the
   global update (matching MLlib's `if (batch.isEmpty()) return this`).
4. Computes `stats_scale = corpus_size / batch_size`, where `corpus_size` is
   `data_rdd.count()` evaluated once at `fit()` start.
5. Pre-multiplies the aggregated sufficient statistics by `stats_scale` and
   passes them to `update_global` as `target_stats`. The raw aggregated stats
   are passed unscaled to `compute_elbo` for ELBO terms representing observed
   evidence.
6. Unpersists the batch RDD before the next iteration.

The per-iteration seed is generated by a `random.Random(random_seed)` instance
seeded once at `fit()` start, mirroring MLlib's `randomGenerator.nextLong()`.

The abstract `VIModel.update_global` parameter is renamed `aggregated_stats →
target_stats` to disambiguate from the raw `aggregated_stats` passed to
`compute_elbo`. Behavior of all existing models is unchanged.

## Alternatives considered
- **Pass `corpus_size` and `batch_size` to `update_global` directly.** Rejected:
  forces every model author to handle scaling, even though the math is identical
  across models. Cleaner to hide it inside the runner.
- **Use `1/fraction` as the unbiased scale factor (avoid `batch.count()`).**
  Considered but rejected: MLlib's realized-batch-size approach gives lower
  per-iteration variance, the cost of `count()` is small once the batch is
  cached, and matching the canonical reference exactly is worth the trivial
  overhead.
- **Without-replacement sampling / epoch traversal.** Rejected: with-replacement
  matches Hoffman et al. 2013 SVI theory and the MLlib precedent, and gives
  i.i.d. samples per iteration as Robbins-Monro assumes.
- **Expose a `stats_scale` kwarg on `update_global` directly.** Rejected after
  deliberation: the runner can do the multiplication, freeing model authors
  from having to think about scaling at all.

## Consequences
- **First introduction of randomness into the framework's run loop.** Runs are
  no longer bit-deterministic by default; reproducibility requires a fixed
  `random_seed`, and even then is best-effort (Spark partition ordering can
  introduce floating-point variation).
- **Contract rename.** `update_global`'s second positional parameter is
  `target_stats` rather than `aggregated_stats`. Behavior unchanged; existing
  callers using positional or `learning_rate=` kwarg are unaffected.
- **Cost.** One extra Spark action (`batch.count()`) per iteration when
  mini-batching is enabled, mitigated by `StorageLevel.MEMORY_AND_DISK` on the
  batch RDD. Negligible vs. the `mapPartitions` work that follows.
- **Robbins-Monro principled regime.** With mini-batching enabled, the
  decaying step-size schedule satisfies its theoretical assumptions; full-batch
  runs (`mini_batch_fraction=None`) remain available but should override `κ`
  if the schedule decays too aggressively. See `RISKS_AND_MITIGATIONS.md`.
